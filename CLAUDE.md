## 3. Protocol Zero -- No AI Attribution

**This is the highest-priority enforcement policy in this repository.**

All output Claude produces must be indistinguishable from human-authored work. No reader -- human or automated -- should be able to identify Claude's involvement from the code, comments, commit messages, or documentation.

### Prohibited Artifacts

- Attribution comments: "generated by", "ai-assisted", "created with AI", "written by Claude"
- Co-author tags: `Co-authored-by:` referencing any AI system
- AI-slop phrasing: "I hope this helps", "as requested", "here is the code", "as an AI"
- Robotic boilerplate: over-explanatory comments restating what code already expresses
- Typographic leakage: em dashes (U+2014), en dashes (U+2013), smart quotes (U+201C/D/2018/19) in source files

### Enforcement Scripts

| Script                       | Purpose                                  | When to Run                        |
| ---------------------------- | ---------------------------------------- | ---------------------------------- |
| `scripts/protocol-zero.sh`   | Scans source for AI attribution patterns | Before declaring any task complete |
| `scripts/check-em-dashes.sh` | Scans source for typographic characters  | Before declaring any task complete |

Both scripts exit 0 on pass, exit 1 on violation. **A violation from either script means the task is not done.** Claude MUST fix all violations before completion.

Hard-exempt paths (never scanned): `docs/`, `.claude/`, `CLAUDE.md`, `*.md` files, `protocol-zero.sh`, `check-em-dashes.sh`.

### Writing Style Rules

- Use ASCII hyphens (`--`), not em/en dashes, in all source files
- Use straight quotes (`"` and `'`), not smart quotes
- Write comments as a human engineer would: terse, technical, no filler
- Commit messages: imperative mood, Conventional Commits format, no AI self-reference

## 5. Workflow Contract

Every task follows this execution order:

1. **Read context.** Check build health from SessionStart hook output. Read relevant source-of-truth docs before touching code.
2. **Identify constraints.** Check which skills, hooks, and scripts apply to the task.
3. **Activate skills.** Use the correct skill for the domain (see Section 8). Multiple skills may apply simultaneously.
4. **Implement.** Write code that satisfies the design spec, design doc, and all applicable skills.
5. **Validate.** Run relevant checks:
   - `scripts/protocol-zero.sh` -- AI attribution scan
   - `scripts/check-em-dashes.sh` -- typographic lint
   - Build verification if Swift files were modified
   - SwiftLint (runs automatically via hook on .swift edits)
6. **Fix violations.** Any hook warning or script failure must be resolved before completion.
7. **Declare done** only when all checks pass and Definition of Done (Section 11) is satisfied.

## 6. Git Workflow

- `main` is protected. All changes reach `main` through a pull request unless Dinesh explicitly overrides.
- Branch naming: `feat/`, `fix/`, `chore/`, `exp/`, `refactor/`, `test/`, `docs/`
- Commit format: `type(scope): imperative description`
- Valid types: `feat`, `fix`, `chore`, `refactor`, `test`, `docs`, `exp`
- `.pbxproj` regenerations are always isolated `chore(project):` commits, never mixed with Swift changes.
- The `commit-message-lint` hook validates format automatically. Fix violations immediately.

## 7. Engineering Standards

- No hardcoded hex colors in Swift. Use `Color("CadenceTokenName")` from xcassets. The `no-hex-in-swift` hook enforces this.
- No force unwraps or unsafe casts without justification.
- No `print()` / `debugPrint()` in committed code. Use structured logging gated behind debug flags.
- No magic numbers. Named constants for all thresholds and tuning values.
- No dead code, stubs, or placeholder comments.
- One concern per file. File name reflects content.
- `project.yml` is the source of truth for Xcode project structure. Never edit `.pbxproj` directly. The `protect-pbxproj` hook hard-blocks any attempt.

## 11. Blocking Conditions

Claude MUST HALT and ask before proceeding when:

- Requirements are missing or ambiguous and correctness depends on them
- Two source-of-truth documents conflict
- A script (`protocol-zero.sh` or `check-em-dashes.sh`) reports violations
- A hook reports a hard-block (exit 2 on PreToolUse, or JSON `{"decision":"block"}` on PostToolUse)
- The SessionStart build health check reports FAILED
- A skill is required but Claude is uncertain which skill applies
- The task would require modifying a locked spec (Design Spec v1.1 is locked)
- Claude cannot validate correctness of its own output

## 12. Definition of Done

A task is complete only when ALL of the following hold:

- [ ] Output aligns with the design spec and design doc
- [ ] All applicable skills were activated and their constraints satisfied
- [ ] `scripts/protocol-zero.sh` exits 0 (no AI attribution artifacts)
- [ ] `scripts/check-em-dashes.sh` exits 0 (no typographic leakage)
- [ ] All hook warnings from the session have been addressed
- [ ] No hardcoded hex values in Swift source
- [ ] No dead code, stubs, placeholders, or AI-slop comments
- [ ] Conventional Commits format on all commits
- [ ] Build compiles if Swift files were modified
- [ ] No unresolved ambiguity
- [ ] No force unwraps without justification
- [ ] This file (`CLAUDE.md`) was followed throughout the task
