# PHASES

**Document version:** 1.0
**Date:** March 2026
**Status:** Approved for planning use
**Authority:** Source of truth for project-level execution sequencing

---

## Purpose

This document is the canonical, top-level execution map for the Cadence iOS application. It translates the design doc and design spec into a deterministic sequence of implementation phases. Every phase represents a bounded unit of work with a single primary objective, clear entry and exit conditions, and explicit dependencies on prior phases.

This document does not contain epics, stories, or tasks. It sits one level above epic planning. When phase definitions are complete, epic decomposition within each phase becomes straightforward without re-deriving project structure.

**How to use this document:**

- Before writing epics for any phase, verify that all declared dependencies are complete.
- When a phase is marked done, the "Completion Standard" section defines what that means.
- If a source document changes, this file must be updated before implementation of affected phases proceeds.
- Phases must be executed in numeric order unless a dependency analysis explicitly demonstrates that parallel execution of a specific pair is safe.

**What this document is not:**

- A sprint plan, a backlog, or a Gantt chart.
- An exhaustive feature list (the design doc and spec fulfill that role).
- A living wish list. Work not implied by the source documents belongs in a separate post-v1 tracking artifact.

## Planning Principles

These principles governed the construction of every phase in this document.

**One phase, one primary objective.** Each phase has exactly one driving purpose. Secondary concerns that could stand alone belong in their own phase.

**Deterministic scope boundaries.** Every phase has an explicit In Scope and Out of Scope column. The boundary is not implied -- it is stated. If work is not listed as in scope for any phase, it is either out of scope for the MVP or missing and must be added with an explicit justification.

**Dependency-first sequencing.** Phases are ordered by their dependency graph, not by feature salience or business priority. No phase begins until its declared dependencies are complete.

**No speculative work.** Phases contain only work that is directly implied by the source documents. No phase includes architecture for hypothetical future requirements, post-beta features, or convenience abstractions.

**No hidden assumptions.** If a phase requires a decision not resolved in the source documents, the Phase Notes section calls it out explicitly.

**Epic-ready granularity.** Each phase is scoped so that a future engineer can write 3-6 epics under it without needing to re-analyze project structure. Phases that would produce fewer than 3 epics are too small and have been merged. Phases that would produce more than 7 epics are too large and have been split.

**Material completion.** A completed phase must leave the project in a materially better state. "Setup" phases are legitimate only when their outputs are prerequisites for multiple subsequent phases.

## Phase Notes

### Phase 0 -- Project Foundation

**Intent:** Establish a working Xcode project that compiles, passes lint, and has all design system color tokens baked into xcassets before any feature code is written.

**Sequencing rationale:** Every subsequent phase adds Swift source files. If XcodeGen project.yml is not correct from the start, each file addition risks project corruption. Color assets must be in xcassets before any UI phase because the `no-hex-in-swift` hook blocks hardcoded hex values -- a chicken-and-egg problem if assets are added later.

**Why separated from Phase 1:** Project foundation is iOS infrastructure. Supabase backend setup is a separate system with a separate failure domain. Conflating them means a Supabase configuration error blocks iOS engineers from starting.

**Likely epics:** XcodeGen project spec + target configuration; xcassets color token setup (10 colors, light + dark); CI workflow skeleton (lint, build jobs); enforcement scripts and hooks (SwiftLint, Protocol Zero, em-dash).

**Known blocker:** `CadencePrimary` token is referenced in Design Spec §7 but undefined in §3. Do not add a placeholder. Block the sharing strip background in Phase 5 until the designer confirms the value.

---

### Phase 1 -- Supabase Backend

**Intent:** Create the live backend against which all iOS-to-server integration will run. No mock server, no local Supabase -- the actual Supabase project.

**Sequencing rationale:** Auth providers must be configured before Phase 2 can integrate them. The partner_connections table must exist before Phase 2's Partner onboarding can validate an invite code. RLS must be correct before any test of partner data access.

**Why separated from Phase 2:** Backend setup is not iOS work. Mixing them introduces a category error in the dependency chain and makes failure diagnosis harder. A backend engineer can execute Phase 1 while an iOS engineer finalizes Phase 0 scaffolding.

**Likely epics:** Supabase project creation + extension enablement; 8-table schema migration; RLS policy set; auth provider configuration (Apple, Google, email/password); Edge Function directory scaffold (full implementation in Phase 10).

---

### Phase 2 -- Authentication & Onboarding

**Intent:** Get a user through the door. Splash -> auth -> role selection -> role-appropriate onboarding -> landing in the correct main shell stub.

**Sequencing rationale:** Authentication produces the JWT that every Supabase call requires. Role selection determines which navigation shell loads. Onboarding writes the cycle_profiles row that seeds prediction input. Nothing else can work without this sequence completing.

**Why separated from Phase 4 (nav shells):** The onboarding path and the main app shell are distinct flows with distinct state machines. Mixing their implementation produces hard-to-debug routing logic. Phase 2 ends when the user lands at the base of either navigation shell -- what they see there is Phase 4 and Phase 9 respectively.

**Likely epics:** Splash screen (CadenceMark.swift, SplashView.swift, animation sequence, reduced motion path); Auth screen UI + supabase-swift session integration; Role selection + AppCoordinator routing; Tracker onboarding form + cycle_profiles write; Partner onboarding invite code entry + validation.

---

### Phase 3 -- Core Data Layer & Prediction Engine

**Intent:** Produce a correct, tested prediction algorithm and local SwiftData schema before any UI reads from them. Prediction correctness cannot be verified once it is buried under UI layers.

**Sequencing rationale:** The prediction engine feeds Phase 5 (Tracker Home), Phase 6 (Calendar), Phase 11 (Reports). All three read prediction data. If the algorithm is incorrect, all three surfaces are wrong. Unit testing the algorithm in isolation, before UI, produces a tighter feedback loop.

**Why separated from Phase 4:** The data layer is not UI work. Prediction logic is algorithmic and testable in isolation. Combining data layer and navigation shell in one phase typically results in both being deprioritized in favor of visible UI.

**Likely epics:** SwiftData model definitions (all 5 entities); prediction algorithm implementation (next period, ovulation, fertile window); confidence scoring (all three thresholds, SD calculation); prediction edge case unit tests (10 required); SyncCoordinator skeleton with write queue structure.

---

### Phase 4 -- Tracker Navigation Shell & Core Logging

**Intent:** Build the primary interaction surface -- the shell every Tracker screen lives in and the Log Sheet that anchors the core daily logging loop. The component library built here (SymptomChip, PeriodToggle, Primary CTA Button) is reused in Phase 5, Phase 9, and Phase 11.

**Sequencing rationale:** Phase 5 (Home), Phase 6 (Calendar), and Phase 11 (Reports) are content within the shell. The shell must exist before content can be placed inside it. The Log Sheet entry point exists in all three -- it must be implemented once, here, and reused.

**Why separated from Phase 5:** The navigation shell is infrastructure. The Home dashboard feed is content. Mixing them conflates two distinct engineering concerns and makes the phase too large (~9 epics combined).

**Likely epics:** 5-tab TabView with Liquid Glass chrome + Log tab modal intercept; Log Sheet content (period toggles, flow chips, symptom grid, notes, private flag, save CTA); SymptomChip component (all states, Sex chip lock icon, isReadOnly parameter); PeriodToggle + Primary CTA Button components; motion spec implementation (chip spring, toggle cross-dissolve, sheet detents, reduced motion gating).

---

### Phase 5 -- Tracker Home Dashboard

**Intent:** Build the primary daily retention surface. A Tracker who opens the app without intending to log should still receive value from the Home feed.

**Sequencing rationale:** Requires both the prediction engine (Phase 3) for cycle status and countdown data, and the shell (Phase 4) for navigation context and chip component reuse.

**Why separated from Phase 6:** The Home dashboard and Calendar are distinct screens with different data access patterns. The Home feed reads current-state predictions; the Calendar visualizes historical data over time.

**Known ambiguity -- Insight Card:** The MVP Spec gives examples of insight content ("You usually experience cramps on days 1 and 2") but does not specify the derivation algorithm. If the insight card requires non-trivial pattern analysis beyond what the prediction engine produces, this must be resolved before Phase 5 implementation begins.

**Likely epics:** Sharing status strip (active/paused states, crossfade, CadencePrimary gap blocker); Cycle Status Card + confidence badge; Countdown Row (both cards, 48pt numerals); Today's Log Card + Insight Card + skeleton loading states.

---

### Phase 6 -- Calendar View

**Intent:** Give the Tracker a visual history of their cycle. The Calendar builds trust through transparency -- the Tracker can see what the app knows and correct it.

**Sequencing rationale:** The Calendar reads period_logs and prediction_snapshots from Phase 3. The day-tap edit path uses the Log Sheet from Phase 4. Both must exist before Calendar implementation is meaningful.

**Why separated from Phase 5:** Calendar is a distinct tab with a distinct data visualization pattern (time-series grid vs. current-state feed). The engineering concerns do not overlap.

**Likely epics:** Month grid layout + day state rendering (all 5 visual states from §12.4); fertile window band + ovulation day indicator; day-tap -> day detail read-sheet (.medium detent); historical entry edit path via Log Sheet (pre-populated).

---

### Phase 7 -- Sync Layer

**Intent:** Make the app live. Convert offline-only SwiftData into a synchronized Supabase-backed system with correct conflict resolution and offline resilience.

**Sequencing rationale:** Sync is a cross-cutting concern, but implementing it before Phases 5, 6, 8, 9 means each of those phases can be implemented with sync already working. Implementing sync after all UI phases means each UI phase is built against a non-live data layer and must be retested after sync is added.

**Why separated from Phase 3:** Phase 3 establishes the SyncCoordinator skeleton specifically to avoid coupling the data schema definition to the full sync implementation. The full sync implementation requires Phase 1 (live Supabase) to be in place.

**Likely epics:** Write queue with offline-first guarantee + initial data pull; last-write-wins conflict resolution + updated_at semantics; Realtime channel subscription setup (partner_connections, daily_logs); NWPathMonitor offline indicator + exponential backoff retry + auth session resilience.

---

### Phase 8 -- Partner Connection & Privacy Architecture

**Intent:** Implement the product's defining feature. Partner sharing with Tracker-controlled granularity, enforced at the database layer, with multiple privacy override mechanisms.

**Sequencing rationale:** Phase 9 (Partner Dashboard) cannot be built without a live partner connection producing data the Partner can read. Phase 8 is the prerequisite. Phase 7 (sync) must be active for permission changes to propagate to Supabase in real time.

**Why separated from Phase 2:** Phase 2 handles the Partner's onboarding code entry screen only -- the connection validation call. The full connection handshake, confirmation screen, and permission management are distinct flows that belong with the feature they enable.

**Likely epics:** Invite code generation + partner_connections write; Partner onboarding code validation + connection handshake + confirmation screen; 6 permission category toggles UI (Settings surface); pause sharing + disconnect flows; privacy enforcement layer (isPrivate before RLS, Sex symptom exclusion, Partner query column projection).

---

### Phase 9 -- Partner Navigation Shell & Dashboard

**Intent:** Build the Partner's complete read-only experience. The Partner experience is not a reduced Tracker experience -- it is a different product with a different IA.

**Sequencing rationale:** Requires Phase 8 (live partner connection producing Supabase-gated data) and Phase 7 (Realtime subscription for live updates). The SymptomChip component from Phase 4 must exist with isReadOnly support before it can be placed in the Partner Bento grid.

**Why separated from Phase 4:** Partner navigation tree is fully isolated from Tracker navigation tree. Implementing them together creates coupling risk at the AppCoordinator level.

**Likely epics:** 3-tab Partner TabView + NavigationStack (isolated from Tracker); Bento grid cards (Phase, Countdown, Symptoms, Notes) using Data Card; paused sharing state (0.25s easeInOut crossfade); Realtime subscription integration + read-only enforcement + empty states.

---

### Phase 10 -- Notifications

**Intent:** Give both users ambient awareness of cycle events without requiring active app opens.

**Sequencing rationale:** Partner notifications require a live partner connection (Phase 8) and a working APNS dispatch path (Edge Function scaffolded in Phase 1). Tracker reminders require reminder_settings table (Phase 1) and sync (Phase 7).

**Known blocker:** Design Spec §15 flags "Notification content specification" as a post-connection flow open item. This specification must be resolved before Phase 10 implementation begins.

**Likely epics:** Tracker reminder scheduling (3 types, UNUserNotificationCenter, configurable advance days); Partner push notification dispatch (3 types, APNS via Edge Function); Edge Function full implementation; notification permission request flow + reminder_settings UI; Partner mute controls.

---

### Phase 11 -- Reports

**Intent:** Surface longitudinal cycle intelligence once the Tracker has sufficient history for it to be meaningful.

**Sequencing rationale:** Reports require at least 2 completed PeriodLog entries to display. The data model from Phase 3 must be populated with real cycle history. Reports cannot be meaningfully tested until that threshold exists.

**Known blocker:** Design Spec §15 flags "Reports screen specification" as a post-alpha open item requiring "2-cycle data model validation". Chart types and metric hierarchy are not fully specified. This must be resolved before Phase 11 implementation begins.

**Likely epics:** Chart components + metric cards (pending chart type specification); reports data reads from SwiftData; 2-cycle minimum gate + empty state.

---

### Phase 12 -- Settings

**Intent:** Provide a coherent configuration surface that surfaces all configuration flows implemented in prior phases.

**Sequencing rationale:** Settings wraps flows from Phase 8 (partner management), Phase 10 (reminder preferences), and Phase 2 (auth session for sign-out). All three must exist before Settings can surface them correctly.

**Why separated from Phases 8, 10:** Settings is a navigation aggregator, not a feature owner. Each feature manages its own data and business logic -- Settings provides the navigation surface. Implementing Settings before its dependencies exist produces a shell with empty placeholder screens.

**Likely epics:** Tracker cycle defaults + partner management section; reminder preferences + app lock (Face ID / Touch ID / passcode); delete all data (local + Supabase, confirmation); Partner Settings (notification preferences, connection status, disconnect, account).

---

### Phase 13 -- Accessibility Compliance

**Intent:** A structured audit pass that verifies every accessibility requirement is satisfied before TestFlight.

**Sequencing rationale:** Accessibility compliance is a cross-cutting concern that applies to every screen and component. Running this pass after all UI phases are complete means one audit covers the full application, rather than repeated partial audits that may miss cross-screen interactions.

**Why separated from feature phases:** Accessibility bugs found in Phase 13 are typically contained to the specific component or layout where the violation occurs. Fixing them does not require architectural changes. This makes Phase 13 a high-confidence gate with well-understood, bounded remediation.

**Likely epics:** Touch target audit + remediation across all interactive elements; VoiceOver accessibilityLabel audit (all chips, lock icon, nav elements); Dynamic Type + reduced motion gating verification; WCAG AA on-device contrast audit (light and dark modes).

---

### Phase 14 -- Pre-TestFlight Hardening

**Intent:** Close all pre-ship open items, validate the CI pipeline, achieve coverage gates, and ship the first TestFlight build to the known beta cohort.

**Sequencing rationale:** Every prior phase must be complete before hardening begins. Hardening applied to incomplete features produces false confidence.

**Why a dedicated phase:** Pre-TestFlight tasks span multiple domains (CI, testing, device validation, asset finalization). Scattering them across feature phases leads to them being deferred. A dedicated phase with a clear completion gate prevents that pattern.

**Likely epics:** CI/CD pipeline (GitHub Actions, Fastlane, secrets, simulator matrix); unit test coverage gate (80%+ on data + domain layers) + UI test suite (Log Sheet, chip toggle, isPrivate); device validation (dark mode contrast, performance, offline E2E); haptic library + CadenceMark Bezier finalization; TestFlight build distribution.

---

## Completion Standard

When all 15 phases (0-14) are complete, the Cadence MVP is complete as defined by the design doc and design spec. The product at that point will be:

- Buildable and passing CI
- Deployable via TestFlight to the known beta cohort
- Functionally complete: auth, Tracker logging, predictions, Tracker Home, Calendar, Partner sharing, Partner dashboard, reports, notifications, settings
- Accessible at WCAG AA
- Privacy-correct with RLS-enforced data access control
- Offline-resilient with durable sync

**Gap handling:** Any feature implied by the design doc or design spec that is not covered by Phases 0-14 is an omission in this document. It must be added as an explicit phase or an explicit addendum to an existing phase, with source references, before implementation. Do not implement it as an undeclared addition to an existing phase.

**Post-v1 work:** Features out of scope for the MVP beta are explicitly listed in MVP Spec §11 (Out of Scope for Beta) and must not be added to any phase in this document. Track them separately.
